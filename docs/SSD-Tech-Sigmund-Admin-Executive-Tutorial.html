Product goals (as implemented in this codebase)
  1. Semi-automatic project updates for the client: feature data and activity logs live in Supabase; the UI subscribes to database changes so dashboards refresh as soon as records change. Optional GitHub-related automation is configurable under admin Settings (webhook processing and log message templates).
  2. Maximum editable access for admins: administrators can bulk-import or update features via CSV, edit client-facing summaries and visibility flags per feature, manage team onboarding, and configure integration settings. Executives receive a streamlined navigation label (“Client Portal”) and cannot reach admin-only routes.

1. How the application is built

The app is a React single-page application built with Vite, using React Router for navigation and Supabase (PostgreSQL + Auth + Realtime + Edge Functions) as the backend.

  • Authentication: Email and password sign-in via Supabase Auth. After login, the app loads the user’s row from the profiles table and reads role, which must be either admin or executive.
  • Authorization in the UI: Routes such as /onboarding, /settings, and /uploads are only mounted for users whose profile role is admin. Executives who attempt those URLs are redirected to the overview.
  • Feature data: Rows in the features table drive Overview metrics, the feature registry, timeline, risks, and team workload. If the database returns no rows or errors occur, the app falls back to bundled seed sample data so the UI remains demonstrable.
  • Phases: Phase definitions used in the Overview and Timeline are currently supplied from static seed data in the codebase (not loaded from Supabase in the current version). Feature rows still carry phaseId and phaseName for grouping and reporting.
  • Real time: The features and team_members hooks subscribe to Supabase Realtime postgres_changes on their tables, then refetch. When any authorized client updates a row, connected browsers refresh automatically without a manual reload.

2. End-to-end workflow

  1. Welcome & project selection: Unauthenticated users land on a welcome screen, choose a project card (for example “SSD-Tech Project Tracker”), then continue to login.
  2. Login: Supabase signInWithPassword. On success, the session is established and the dashboard loads.
  3. Role resolution: AuthProvider fetches profiles.role. That role drives sidebar items (admin-only links) and page-level behavior (for example whether clicking a feature row opens the editor).
  4. Daily operations: Admins maintain features (UI edits and/or CSV uploads), optional GitHub-driven log entries (when wired on the server side), and team membership. Executives consume Overview, Features, Timeline, Team, and Risks for status and delivery confidence.
  5. Audit trail: When an admin saves certain fields through the feature editor, the app inserts a row into update_logs describing the change at a high level. Per-feature activity is shown in the Features detail panel and can also receive github_push type entries when your backend automation inserts them.

3. Executive (client) guide

This section is written for client executives and steering stakeholders who need a trustworthy, current picture of delivery without operating internal tools.

3.1 What you can rely on

  • Executive Overview: Portfolio-level metrics (phases, completion counts, risk counts, testing load, assignee breadth, overall completion percentage derived from average feature progress). A LIVE indicator on the overview communicates that the view is tied to live data refresh behavior.
  • Major Deliverables: Highlights features that have an MVP link, SRS reference, and/or executive summary, with status badges (on track, delayed, completed, and so on).
  • Phase Overview: Per-phase completion bars and counts of in-progress, completed, and delayed features.
  • Feature Tracker: Searchable, filterable table of features (phase, stage, on-track status). Click a row to read details: owner, priority, description, executive summary, dependencies, deadline alert chips, and links to MVP demos where configured.
  • Project Timeline: Milestone-oriented view: current focus, filters (in progress, completed, blocked, at risk), expandable cards with tasks, summaries, blockers, and dependencies; sidebar stage distribution and active risks.
  • Team: Roster with current assignment snapshot, stage, due date, next task, and capacity badge (available, near capacity, overloaded).
  • Risks & Blockers: Concentrated view of items that are not fully on track or have deadline or staleness alerts, with severity styling and escalation-oriented panels (some sidebar actions are presentational in the current UI).

3.2 What executives do not see (by design)

  • Internal-only fields: GitHub pull request URLs, internal developer notes, and raw blocker text in some contexts are restricted to administrators so the client narrative stays clean while delivery teams retain context.
  • Admin navigation: Onboarding, Uploads, and Settings are hidden from the executive sidebar.
  • Inline editing: Executives do not open the modal feature editor; they read curated fields prepared by the delivery organization.

3.3 How to use the portal effectively

  1. Start at Overview each week for the portfolio pulse, then drill into Timeline for sequencing and Risks when delivery confidence is the question.
  2. Use Feature Tracker filters when you care about a single phase or a specific delivery stage (for example everything in testing).
  3. Open MVP / Demo links when provided to validate tangible progress against expectations.
  4. Read Executive summary text on features and deliverable cards first; it is the concise client-facing storyline maintained by admins.

4. Administrator guide

This section is for project administrators, PMs, and delivery leads responsible for accuracy, client messaging, and access control.

4.1 Navigation exclusive to admins

  • Onboarding: Invite users and manage the active team list. Invitations are executed through the Supabase Edge Function invite-member, which requires the caller to be an admin, creates or invites the auth user, sets profiles.role, and upserts team_members.
  • Uploads: Bulk import or update features from CSV. Parsed rows are upserted into features on feature_id. Use this for sprint planning imports, replanning waves, or reconciling spreadsheets with the database.
  • Settings: GitHub webhook automation toggles and the activity log message template (placeholders {sha}, {message}, {url}). This supports semi-automatic progress narration when your repository integration writes matching update_logs rows (SRS-tagged commits as described in the in-app copy).

4.2 Feature maintenance (single-item editor)

On Features, admins click a row to open the Edit modal. Saved fields (mapped to Supabase) include:

  • Progress (0–100): Drives completion rings, bars, and summary averages; also feeds on-track classification together with deadlines and status.
  • Assigned developer: Dropdown sourced from team_members. Ensure roster names align with how you assign work in feature rows for consistent Team page linkage.
  • Executive summary: Short client-facing status; appears on Overview deliverable cards, feature details, timeline, and risk cards.
  • MVP / demo URL: External demo link surfaced to executives.
  • SRS requirement ID: May be a plain id (shown as a badge) or a full URL (rendered as a document link on admin-heavy views).
  • GitHub PR URL and internal notes: Visible to admins only in relevant views.
  • Visible in Executive view: Checkbox mapped to client_visibility. Admins use this to hide internal or sensitive work items from client-facing curation workflows. Note: ensure your organization’s policy for whether downstream pages filter by this flag is aligned with how you use the database (the field is stored and editable for governance and future filtering).

Each successful save from this path also appends an audit-style row to update_logs listing which fields changed.

4.3 Bulk CSV workflow (Uploads page)

  1. Click Choose CSV File or paste CSV into the text area. Maximum file size enforced in UI: 2 MB.
  2. Required columns match the parser: feature identity, scheduling, ownership, stage, status, progress, tasks, QA and design/dev status strings, last update metadata, and clientVisibility as the string true or false.
  3. Optional columns include revised deadline, on-track status, executive summary, URLs, and internal notes.
  4. Use Copy Template Headers to start a valid sheet. Dependencies in CSV use pipe | separators.
  5. Apply Upload runs validation; confirm the modal to upsert. Existing IDs are updated, new IDs are inserted.

4.4 Team onboarding and removal

  • Fill name, email, and role, then Send Invite. The Edge Function validates that only an admin can invite and that the invited portal role is one of the allowed values enforced server-side: admin or executive (these control dashboard privileges and are stored on profiles.role).
  • Remove deactivates a member in team_members (active: false); the list shows active members only.

Deployment note: If your onboarding form collects a job title (for example “Frontend Developer”) but the Edge Function expects a portal role (admin | executive), invitations will fail validation until the UI and API agree on one payload shape—for example a dedicated “Portal access” selector for invitees plus a separate job-title field stored only on team_members.

5. Curating the executive view and delivery logs

Your operational goal is a semi-automatic client display: numbers and statuses update in real time from the database, while humans control the narrative and boundaries.

5.1 Narrative layer

  • Executive summary per feature is the primary prose control for what executives read in multiple surfaces.
  • Progress and deadlines should be kept honest; the app computes on-track status from rules combining completion, blocked/delayed flags, and proximity to deadlines.
  • Activity History on the feature detail panel shows the last several update_logs entries with author display name and timestamp—useful for demonstrating cadence to the client alongside summaries.

5.2 Automation layer (optional)

  • Configure GitHub integration messaging under Settings so automated inserts into update_logs read consistently (for example commit SHA and message substitution).
  • When developers include SRS tags in commit messages (per in-app guidance), server-side automation can attach those events to the correct feature for traceability from code motion to client-visible history.

5.3 Organizational layer

  • Use CSV uploads for batch realignment after planning meetings.
  • Use client visibility to keep internal-only features in the system without exposing them in client-facing storylines, according to your delivery policy.

7. Data model and live updates (reference)

  • profiles — user id, role (admin | executive), display name.
  • features — full delivery record for each feature; primary driver of dashboards.
  • update_logs — per-feature audit and automation trail (manual | github_push).
  • team_members — roster for assignments and onboarding.
  • system_settings — key/value store; GitHub integration JSON under github_integration.

Realtime: Subscriptions on features and team_members refetch on change. update_logs uses a filtered channel per selected feature so detail panels stay fresh.
